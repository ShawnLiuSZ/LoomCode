## 1. R-2 上下文截断（P0-3）—— ✅ 闭环（方案更优）

Round 1 的问题：partition 被填充但请求仍发 `a.messages`，且无截断。

**本轮做法**（比原建议更干净）：**直接删除那套 inert 的 partition 机制**，改为真实截断。

| 检查点 | 结果 | 证据 |
|---|---|---|
| 死机制是否清除 | ✅ | `internal/agent/` 已无任何 `partition`/`contextpkg` 引用；`loop.go` 不再 import `internal/context` |
| 截断是否实现并接入 | ✅ | `loop.go:96` `truncateMessages`，在每次请求前调用（流式 `:145`、goal 路径同样） |
| 截断策略是否合理 | ✅ | 留 system(idx 0) + 最近 4 条，优先删最旧 `tool` 结果，预留 20% 输出预算（`loop.go:101-128`） |
| 长会话崩溃隐患 | ✅ 已消除 | 超出 80% 窗口即裁剪，不再无界增长 |
| 前缀缓存收益 | ✅ 无实质损失 | 靠 provider 端对稳定 system 前缀的自动缓存 + 已修的 cached 计费 |

**收尾结论**：P0-3 关闭。partition 机制的删除是正确的止损——比"为接而接"更诚实。

### 1.1 遗留细节（非阻塞，建议硬化）

| 等级 | 项 | 说明 |
|---|---|---|
| P1 | **截断可能产生非法消息序列** | `truncateMessages` 优先删 `tool` 消息（`loop.go:116`）。若删掉某 `tool` 响应却保留其对应的 assistant `tool_calls`，OpenAI 类严格接口会返回 400（tool_call 无响应）。建议：删除时按 assistant+其 tool 响应**成对**删除，或删 assistant tool_calls 时连带删其 tool 结果。 |
| P2 | token 估算偏粗 | `estimateTokens` 用 `len/3`（`loop.go:86`），仅作安全护栏够用；如需精确成本控制可接真实 tokenizer。 |
| P2 | 未知模型不截断 | `getContextWindow` 找不到模型返回 0 → 不截断（`loop.go:79`）。OpenAI 兼容自定义模型若未在配置写 `ContextWindow`，则失去保护。建议配置层强制或给默认值。 |

## 2. R-1 CI / 版本对齐 —— 🟡 基本闭环，留一个尾巴

Round 1 的问题：`go.mod 1.25.0` vs CI 矩阵 `1.21–1.23`，会让 CI 崩。

| 检查点 | 结果 | 证据 |
|---|---|---|
| 硬矛盾是否消除 | ✅ | `go.mod:3` 降到 `go 1.23`，去掉 `.0` patch 级 pin |
| CI 是否会因版本崩 | ✅ 不会 | `1.23` job 正常；其他 job 也用 `1.23`（`ci.yml:45,62`） |
| 矩阵是否真实 | ⚠️ **否** | 矩阵仍含 `1.21`/`1.22`（`ci.yml:14`），低于 go.mod 的 `1.23` 要求 |

**尾巴说明**：跑在 Go 1.21/1.22 的 job 会被 `GOTOOLCHAIN=auto`（默认）静默下载并切到 1.23 执行——CI **不会红**，但"在 1.21/1.22 上验证过"是假象，矩阵失去意义。

**建议（二选一）**：
- 矩阵收敛到 `['1.23','1.24','1.25']` 这类 `≥ go.mod` 的版本；
- 或确有支持 1.21 的需求，则把 `go.mod` 真正降到 `go 1.21` 并验证不用 1.22+ 的 API。
