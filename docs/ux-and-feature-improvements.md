# Helix 体验优化提案（产品 / TUI 视角）

> 把 Helix 当作**终端里的编程 agent**来评判，而不是从功能清单出发。
> 本文是带明确取舍和优先级的改进建议，不是审计/验收记录。

---

## 核心判断

编程 agent 好不好用，不该用"功能多不多"衡量，而是两件事：

1. **核心编程闭环**：让 agent 改代码 → 我能看懂并信任这个改动 → 跑起来验证。这条顺不顺，决定它是不是一个**能用的编程工具**。
2. **打字 / 读输出的体验**：输入框顺不顺手、输出读不读得下去。这条决定它**用着累不累**。

现状的精力错配：很多投入花在"造能力"（分布式协作、语义索引、Web Dashboard…），但真正决定日常体验的是上面这两条——而它们恰恰还薄。**下面的优先级都围绕这两条排。**

---

## 一、TUI 怎么更友好（投入产出比最高）

按建议动手顺序排列。前三项基本是"用成熟组件替换手写实现"，便宜且体感巨大。

| # | 改进 | 为什么 | 怎么做 | 体感 |
|---|---|---|---|---|
| 1 | **输入框换 `textarea`** | 现在手写输入：不能用方向键、不能改中间、无历史、粘贴多行代码会被逐行触发发送。编程场景天天要粘代码、写多行 prompt | 换 Bubbles `textarea`：多行（Shift+Enter 换行 / Enter 发送）、光标编辑、`↑↓` 调历史、括号粘贴 | ⭐ 单点最大改善，"难用"→"顺手" |
| 2 | **输出渲染 markdown + 代码高亮** | agent 回复 90% 是带代码块的 markdown，纯文本铺出来可读性差一个量级 | glamour/chroma 渲染；user/assistant/tool 三角色用颜色/前缀区分 | 阅读体验质变 |
| 3 | **可滚动 viewport** | 长输出读不了历史是硬伤 | Bubbles `viewport`，支持鼠标滚轮 + PgUp/PgDn | 基础但必需 |
| 4 | **可见的 agent 活动 + 可中断** | 多步工具循环时只有干转的 spinner，用户不知道它在干嘛 | 状态行实时显示"第 3 步 · 🔧 运行 `go test`"；Esc 随时打断（已具备） | 可见 + 可中断 = 信任 |
| 5 | **常驻状态栏** | 长会话需要随时掌握成本和上下文余量 | 显示 `模型 · 模式 · 上下文 12k/128k · 本轮 $0.03`；快满时预警（配合截断功能） | 心里有数，不被静默截断吓到 |
| 6 | **`/` 命令面板 + `@文件` 补全** | 命令难发现；指定文件进上下文是编程 agent 标配 | `/` 弹模糊搜索带说明的菜单；`@path` 从文件树补全 | 对齐 aider `/add`、Claude Code `@` |

## 二、功能层面优化（产品层）

### 1. 编辑 / Diff 审查闭环 —— 命门，排第一

agent 要改文件时，给用户一个**彩色 diff + 接受/拒绝/编辑**的确认流（即 SEARCH/REPLACE → `/apply` 的思路）。

- 这是 aider / Claude Code 体验最核心的地方。
- **如果这块弱，TUI 再漂亮也不算一个好用的编程 agent。** 其他一切都服务于"它能不能改好我的代码、我敢不敢信这个改动"。

### 2. 上下文管理 —— 真正的战场

当前按字节粗暴截断，方向不对。应做成：

- **`@file`**：显式把文件塞进上下文，用户掌控。
- **repo-map**：用 tree-sitter 给整个仓库做结构地图，agent 不用读全部文件就知道代码结构 —— **同时提质量和省钱**。
- **接真 embedding**：现有语义索引用的是 MockEmbeddings（假的），接上真 embedding provider 才能"自动找相关文件"。
- **`/compact`**：上下文快满时让用户主动总结压缩历史，而不是静默丢消息。

### 3. 友好权限 + git 原生

- **权限别只有"拦/放"**：给"本次允许 / 本会话总是允许 / 拒绝 / 改一下"四个选项。
- **git 原生**：改完自动生成 commit message 提交、随时 `diff against HEAD`。编程 agent 活在 git 里。

### 4. 会话可续可搜 + 扩展

- 已有 SQLite FTS5，正好拿来做"**搜索过去的对话**"和 `/sessions` 选择器。
- **MCP 接线**：让用户挂自己的工具（连数据库、浏览器、项目专用工具）—— 现代扩展性标配。

## 三、推荐执行顺序

```
第一波：TUI 脱胎换骨（约 1 周，全在 internal/ui）
  textarea 输入框  →  markdown/高亮渲染  →  可滚动 viewport
        ↓
第二波：让它真正能改代码
  编辑/Diff 审查闭环  +  友好权限
        ↓
第三波：让它改得准、省钱
  @file  →  repo-map  →  真 embedding  →  /compact
        ↓
第四波：续航与扩展
  会话搜索/续接  +  git 自动提交  +  MCP 接线
```

**明确先放一放**（不改变日常体验，性价比低）：分布式 Agent 协作、Web Dashboard、桌面客户端 / VSCode 扩展。等核心闭环和 TUI 稳固后再说。

---

## 四、第一步建议

从 **`textarea` 输入框** 起步：

- 改动局限在 `internal/ui`，风险小、不碰核心逻辑；
- 体感提升最直接 —— 做完立刻能感觉到不一样；
- 是后续所有 TUI 改进（历史、粘贴、多行）的地基。
